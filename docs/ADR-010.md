# ADR-010 : Résilience des consommateurs de notifications

---

## Statut

Accepté

---

## Contexte

Disruption #2, message 1 : un matin, l'API email tombe. Avec l'implémentation issue de la Disruption #1, le `Dispatcher` itère déjà sur les canaux et ne retourne pas l'erreur du canal en panne — un bon début. Mais deux exigences nouvelles s'ajoutent :

- l'échec d'un canal doit être **persisté** pour retraitement manuel ;
- on doit pouvoir **simuler une panne** d'un canal pendant la démo, sans recompiler ni redéployer, pour prouver l'isolation ;
- les autres consommateurs (`Realtime`, `Audit`) ne doivent ressentir aucun effet de bord : ils sont déjà des handlers indépendants sur l'`EventBus`, donc cette exigence est satisfaite par construction — il faut uniquement empêcher le canal email cassé de propager une erreur au handler `notification` qui les regroupe.

L'existant : `Dispatcher.Dispatch` swallow déjà les erreurs avec un `log.Printf` et continue. Manque la persistance et l'injection de panne.

---

## Décision

Nous avons décidé d'introduire trois éléments :

1. **Une dead-letter persistée dans la même base relationnelle** : table `failed_notifications` avec status `pending`/`retried`, exposée par l'interface `FailedNotificationRepository`. Le `Dispatcher` y enregistre tout échec de canal, sans bloquer ni propager.

2. **Un décorateur `FaultInjectingChannel`** dans la couche infrastructure, qui implémente l'interface `Channel` du domaine. Il wrappe un canal existant (ici `EmailChannel`) et expose `SetFailing(bool)`. Quand actif, son `Send` retourne directement une erreur sans appeler le canal sous-jacent. C'est un **pattern Decorator** : le `Dispatcher` ne sait rien de l'existence du wrapping.

3. **Une API admin** sous `/api/v1/admin/notifications/...` permettant de basculer le toggle, lister les messages en échec, et déclencher un retry synchrone manuel. Cette API est protégée par le JWT existant (par défaut, tout user authentifié a accès — voir Vigilance).

Le retry est **synchrone et manuel** pour cette livraison : pas de worker en arrière-plan, pas de back-off exponentiel. C'est un raccourci assumé documenté.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| Retry automatique avec back-off exponentiel | Demande un scheduler ou un worker ; introduit une concurrence à tester. Hors scope d'une livraison de 2 h. À introduire avec une queue (NATS, Kafka) en production. |
| Stocker les échecs dans un fichier append-only sur disque | Plus simple mais pas requêtable, pas survivable à un restart de container, pas observable depuis l'UI. La même base PostgreSQL fait l'affaire. |
| Faire échouer le canal via une variable d'environnement et redémarrer | Casse la démo : il faudrait redémarrer le container pour changer de mode. Le décorateur permet un toggle live, donc une démonstration plus convaincante. |
| Throw l'erreur du canal et la gérer dans le handler appelant | Viole le contrat « les autres canaux continuent ». Le `Dispatcher` doit absolument swallow pour ne pas casser sa boucle. |
| Mettre à jour `Channel.Send` pour retourner un wrapper `Result` au lieu de `error` | Trop intrusif pour un gain marginal. Le `error` Go suffit pour propager le statut. |

---

## Conséquences

**Positives :**
- Aucun changement aux services métier (`TaskService`, `ProjectService`) ni aux trois autres consommateurs (`Realtime`, `Audit`, `ConsoleHandler`). Vérifié par `grep -r "fault\|failed_notification" internal/task internal/project` → 0.
- Le décorateur peut être appliqué à n'importe quel canal sans le modifier — Slack, Teams, SMS pourront aussi être wrappés à l'avenir.
- Les opérateurs peuvent observer les pannes via l'UI `/admin/notifications` et les rejouer en un clic.
- La table `failed_notifications` est aussi un **audit naturel des incidents** : on sait combien de tentatives, l'erreur exacte, l'horodatage.

**Négatives / compromis :**
- Le retry est synchrone et bloque la requête HTTP `/retry` jusqu'à la réponse du canal. Acceptable car appelé manuellement, sur un message à la fois.
- Pas d'idempotence côté canal : un retry envoie potentiellement un email déjà envoyé si le premier échec est faux-négatif. Pour de l'email réel, il faudrait un identifiant idempotent côté provider — non pertinent tant que l'email reste un log.
- Le statut `retried` est posé à la première réussite : si un message est rejoué plusieurs fois (succès puis nouvelle panne), seul le premier succès est tracé. À affiner avec un historique de tentatives en production.

**Points de vigilance :**
- L'endpoint admin n'a **pas** de rôle/permission : tout user authentifié peut basculer le canal email en panne. C'est acceptable pour le pilote mais dangereux en production. À corriger avec un rôle `admin` (ADR ultérieur).
- Le décorateur garde l'état en mémoire : un redéploiement remet le canal en marche. C'est plutôt un bug-feature, mais à savoir pour la démo.
- La table `failed_notifications` peut grossir vite si la panne dure ; pas de TTL ni de purge automatique. Prévoir une politique de rétention en production.
