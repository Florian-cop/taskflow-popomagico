# ADR-008 : Audit trail en consommateur d'events

---

## Statut

Accepté

---

## Contexte

Pour répondre à "qui a fait quoi, sur quelle entité, à quelle heure ?", toute écriture sur `Task` et `Project` doit être tracée. Cette traçabilité ne doit pas polluer la logique métier, et le backend de stockage (base relationnelle, service dédié, fichiers) doit être interchangeable.

---

## Décision

Nous avons décidé de créer un bounded context `audit` qui est un **pur consommateur d'events**. Il s'abonne à tous les events métier (`user.created`, `project.created`, `member.added`, `task.created`, `task.moved`, `task.assigned`) sur l'`EventBus`.

Pour chaque event reçu, le handler :
- lit le `userID` depuis le `context.Context` (injecté par le middleware JWT, voir ADR-005)
- extrait `EventName`, `AggregateID` et l'`AggregateType` (déduit du préfixe du nom d'event : `task.moved` → `task`)
- sérialise l'event complet en JSON dans le champ `payload`
- persiste via `AuditRepository` (interface)

La seule implémentation actuelle est `GormAuditRepository` → table `audit_logs`. Changer de backend = implémenter `AuditRepository` ailleurs (client Elastic, fichier append-only, service dédié gRPC) et changer une ligne dans `main.go`.

**Les services `TaskService` et `ProjectService` n'importent rien de `audit/`.** Un `grep -r "audit" internal/task internal/project` doit renvoyer zéro résultat. C'est le critère d'acceptation.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| Appeler `AuditService.Record()` depuis chaque service métier | Pollue la logique métier et viole la contrainte du PO. |
| Hooks GORM (`AfterUpdate`) sur les modèles | Couple l'audit à l'ORM et rend la règle invisible depuis le domaine. Pas de `userID`. |
| Triggers SQL côté PostgreSQL | Perd la traçabilité de l'acteur métier (les triggers voient uniquement la connexion BDD). |
| Envoyer à Kafka/Elastic directement depuis les services | Couplage technique fort — justement ce que l'event bus évite. |

---

## Conséquences

**Positives :**
- Aucune modification des services métier. Le tag `audit` peut être retiré demain sans casser quoi que ce soit.
- Ajouter un event à tracer = une ligne dans l'appel `auditHandlers.Register(...)` de `main.go`.
- Payload JSON complet : suffisant pour reconstituer le contexte d'une action a posteriori.

**Négatives / compromis :**
- Si le handler d'audit échoue, l'event est consommé mais non tracé (l'`InMemoryEventBus` logge l'erreur et continue). Acceptable pour le pilote ; en production, il faudrait un bus persistent avec retry (ADR ultérieur).
- La sérialisation JSON des events utilise leurs champs exportés ; les events domaine actuels ont des champs non-exportés — le payload résultant est donc vide pour certains events. À corriger en ajoutant une méthode `MarshalJSON()` sur les events, ou en exposant les getters.

**Points de vigilance :**
- Le `userID` peut être vide (événements CLI, événements système). Bien documenter ce cas dans la lecture des logs.
- La table `audit_logs` va croître — prévoir une politique de rétention/archivage en production.
