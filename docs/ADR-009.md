# ADR-009 : CLI d'administration comme second adaptateur entrant

---

## Statut

Accepté

---

## Contexte

Le Product Owner demande une CLI pour créer des projets, des tâches, et générer un jeu de démo. L'exigence de durabilité est explicite : **pas de duplication de la logique de création**. Si l'ajout de la CLI oblige à réécrire la logique métier, c'est un signal d'alerte architectural.

---

## Décision

Nous avons décidé de créer un second binaire `cmd/cli/main.go` qui câble **exactement les mêmes services applicatifs** (`ProjectService`, `TaskService`) que `cmd/api/main.go`.

La CLI est un **adaptateur entrant** au même titre que l'API REST : elle parse les arguments `os.Args` avec `flag`, construit un `CreateProjectDTO` ou `CreateTaskDTO`, et appelle le service. Le service ignore qui l'appelle — HTTP, CLI, tests, peu importe.

La commande `generate-demo` crée un projet + 9 tâches (3 TODO, 3 IN_PROGRESS, 3 DONE). Les tâches IN_PROGRESS / DONE sont produites en enchaînant `MoveTask` — donc les transitions domaine (`TaskStatus.CanTransitionTo`) sont respectées, et les events correspondants sont publiés. Les consommateurs (console, audit, notifications) fonctionnent pendant l'exécution CLI exactement comme via l'API.

Aucune librairie CLI externe (cobra, urfave-cli) : `flag` + `switch` sur `os.Args[1]` suffisent pour 3 commandes. À réévaluer si la CLI grossit.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| Un endpoint REST `/admin/generate-demo` | Mélange les préoccupations : l'API est pour les end-users. |
| Un script SQL de seed | Court-circuite le domaine : les règles de transition, les events, les invariants ne sont pas exercés. Exactement ce qu'il ne faut pas faire. |
| Réimplémenter la création de projet en CLI | Dupliquerait la logique — révélateur d'une mauvaise séparation. |
| Cobra / urfave-cli | Dépendance injustifiée pour 3 sous-commandes simples. |

---

## Conséquences

**Positives :**
- Zéro duplication : l'ajout d'une règle métier dans `TaskService` se propage automatiquement à la CLI.
- Les events sont publiés par la CLI → auditable et observable.
- `docker compose exec api /cli generate-demo` permet de seeder la stack sans installer Go localement.

**Négatives / compromis :**
- La CLI ouvre sa propre connexion BDD (pas de réutilisation avec l'API en cours). Acceptable — c'est un outil d'admin ponctuel.
- Pas de userID dans le contexte des commandes CLI : les audits auront `userID=""`. Documenté ; à corriger en injectant un `WithUserID(ctx, "cli-admin")` si besoin.

**Points de vigilance :**
- Les deux binaires doivent rester cohérents dans les imports (même version des services). Vérifié à chaque `go build ./...`.
- Si la CLI grossit (>5 sous-commandes), passer à cobra avant que le `switch` ne devienne illisible.
