# ADR-007 : Notifications multi-canaux via Strategy et bus d'events

---

## Statut

Accepté

---

## Contexte

Il faut notifier les membres sur `task.assigned` (assignee) et `task.moved` (tous les membres du projet). Les canaux initiaux sont l'email et l'in-app. Slack, Teams et SMS sont prévus dans les 6 mois. Chaque utilisateur doit pouvoir activer/désactiver les canaux via une API.

Contrainte non négociable : **ajouter un nouveau canal ne doit jamais modifier les services qui déclenchent les notifications** (`TaskService`, `ProjectService`).

---

## Décision

Nous avons décidé de créer un bounded context `notification` organisé autour d'une interface `Channel` (pattern Strategy) :

```go
type Channel interface {
    Name() string
    Send(ctx context.Context, n *Notification) error
}
```

Le `Dispatcher` reçoit une `[]Channel` et, pour chaque notification, consulte les préférences de l'utilisateur puis appelle les canaux activés. Ajouter Slack = écrire `SlackChannel{}` dans infra et l'ajouter à la slice dans `main.go`. Rien d'autre ne change.

Les event handlers du bounded context s'abonnent aux events métier sur l'`EventBus` existant. Les services métier ne connaissent pas la notification : c'est un **consommateur pur**.

Pour éviter que le bounded context `notification` dépende de `project`, nous avons introduit l'interface `MemberFinder` (côté notification) implémentée par un adaptateur `ProjectMemberFinder` (infra) qui wrap le `ProjectService`. Le coupling cross-context est isolé à la couche infra.

L'`EmailChannel` est un **raccourci assumé** : il loggue `[EMAIL] to=... title=... body=...` au lieu d'envoyer un vrai SMTP. À remplacer en production par un client SendGrid/SES.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| Appels directs depuis `TaskService.MoveTask` | Viole la contrainte du PO : ajouter Slack forcerait à modifier le service métier. |
| Une table `user_notifications` par canal | Multiplication du schéma BDD à chaque canal ; préférer une table unique + préférences sérialisées. |
| Un vrai SMTP dès maintenant (Mailhog local) | Over-kill pour le pilote ; le log console suffit pour valider l'architecture. |
| Préférences en fichier JSON local | Pas multi-instance, pas migrable. On utilise la même BDD PostgreSQL. |

---

## Conséquences

**Positives :**
- Ajout d'un canal = 1 fichier `xxx_channel.go` + 1 ligne dans `main.go`. Zéro diff dans `task/`, `project/`.
- Les tests unitaires du Dispatcher utilisent des mocks `Channel` triviaux.
- Les préférences sont centralisées dans `notification_preferences` avec un JSON flexible (clé = nom du canal), donc ajouter un canal n'impose aucune migration SQL.

**Négatives / compromis :**
- L'EmailChannel ne fait que logger : un utilisateur pourrait s'attendre à recevoir un vrai email. Documenté dans l'API/README.
- Le routage `task.moved` → N notifications n'est pas groupé (N membres = N inserts) — acceptable à faible échelle, à batcher plus tard.

**Points de vigilance :**
- Le `ProjectMemberFinder` crée un coupling notification → project. Acceptable en infra, **interdire** ce type d'import depuis les domaines respectifs.
- Bien checker les préférences **avant** d'envoyer — un bug ici fuit des mails non désirés.
