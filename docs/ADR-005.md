# ADR-005 : Stockage de l'audit trail

---

## Statut

Accepté

---

## Contexte

TaskFlow doit tracer toute action d'ecriture sur les entites `Task` et `Project` pour repondre a la question : "Qui a fait quoi, sur quelle entite, et a quelle heure ?".

Cette tracabilite doit etre decouplée de la logique metier : les services de gestion de taches et de projets ne doivent pas etre modifies pour integrer l'audit. De plus, le backend de stockage des logs d'audit est amene a changer (base relationnelle, service dedie, fichiers) sans impact sur le domaine.

---

## Decision

Nous avons decide de stocker les entrees d'audit dans une table PostgreSQL `audit_log`, accessible via une interface `AuditRepository`.

Un `AuditHandler` ecoute les domain events publies sur le bus interne (`task.created`, `task.moved`, `task.assigned`, `project.created`, `member.added`) et construit une entree d'audit contenant :
- **actor** : identifiant de l'utilisateur a l'origine de l'action
- **action** : type d'evenement (`created`, `moved`, `assigned`, etc.)
- **entity** : type d'entite concernee (`Task`, `Project`)
- **entityId** : identifiant de l'entite
- **timestamp** : horodatage de l'action
- **details** : donnees complementaires (ancien statut, nouveau statut, etc.)

Le handler depend uniquement de l'interface `AuditRepository`.

---

## Alternatives envisagees

| Alternative | Raison du rejet |
|-------------|-----------------|
| Fichiers JSON / logs structures (Winston, Pino) | Pas de requetage simple pour retrouver l'historique d'une entite. Adapte au monitoring, moins a l'audit metier. |
| Service externe dedie (Elasticsearch, Datadog) | Surcharge operationnelle injustifiee a ce stade. Necessite une infrastructure supplementaire et une gestion de disponibilite. |
| Audit directement dans les services metier | Viole le principe de decouplage. Chaque service devrait connaitre le mecanisme d'audit, rendant les tests plus complexes et l'evolution plus couteuse. |

---

## Consequences

**Positives :**
- Aucune modification des services metier : l'audit est un handler comme un autre sur le bus d'evenements.
- Requetage SQL simple pour consulter l'historique d'une entite ou d'un utilisateur.
- Le remplacement du stockage (ex. migration vers Elasticsearch) se limite a creer un nouvel adaptateur implementant `AuditRepository`, sans toucher au handler ni au domaine.
- Testable independamment : on peut verifier que le handler produit les bonnes entrees sans base de donnees.

**Negatives / compromis :**
- Les entrees d'audit partagent la meme base que les donnees metier, ce qui peut poser des problemes de volume a grande echelle.
- Le traitement est synchrone via le bus interne : un ralentissement de l'ecriture audit peut impacter le temps de reponse. Acceptable pour les volumes actuels.

**Points de vigilance :**
- Ne jamais appeler `AuditRepository` directement depuis un service metier. Toujours passer par le bus d'evenements.
- Prevoir un index sur `entityId` et `timestamp` pour les requetes d'historique.
- Documenter la migration vers un stockage externe quand les volumes le justifieront.

**Evolution production :**
- Remplacer `OrmAuditRepository` par un adaptateur Elasticsearch ou un service d'audit SaaS.
- Passer le handler en traitement asynchrone (queue) si les volumes deviennent critiques.
