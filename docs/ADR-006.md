# ADR-006 : Temps réel Kanban via adaptateur sortant WebSocket

---

## Statut

Accepté

---

## Contexte

Quand un membre déplace une tâche, tous les autres membres du projet doivent voir le changement sans recharger. Le Product Owner précise que la technologie temps réel sera probablement changée dans 6 mois selon les contraintes d'infra, et que les mises à jour doivent être **scopées par projet** (un membre du projet A ne reçoit pas les events du projet B).

---

## Décision

Nous avons décidé d'introduire un bounded context `realtime` minimal dont le domaine ne contient qu'une interface : **`Broadcaster.Broadcast(ctx, projectID, []byte)`**.

L'implémentation concrète `WSBroadcaster` (gorilla/websocket + `sync.Map` indexée par projectID) vit dans `internal/realtime/infrastructure/`. Aucun service métier (`TaskService`, `ProjectService`) ne la connaît.

Un **event bridge** (`internal/realtime/application/event_bridge.go`) s'abonne aux events métier (`task.created`, `task.moved`, `task.assigned`, `member.added`) sur l'`EventBus` existant, extrait le `ProjectID` via une interface `ProjectAware` satisfaite implicitement par les events concernés, sérialise en JSON et appelle le broadcaster. Le service task ne bouge pas d'une ligne.

Le handler HTTP (`/api/v1/projects/{id}/ws`) vérifie le membership via `ProjectService.GetProject` avant d'accepter l'upgrade WebSocket. L'authentification passe par query param `?token=...` (raccourci assumé pour compat navigateur — les WebSocket browser ne permettent pas de header Authorization custom).

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| Server-Sent Events (SSE) | Unidirectionnel et pas encore universellement supporté derrière certains proxies. Gardé comme plan de repli. |
| Centrifugo / Pusher managés | Introduit une dépendance tierce coûteuse pour ce pilote. Peut être branché plus tard en remplaçant l'impl de `Broadcaster`. |
| Polling HTTP (5s) | Latence inacceptable pour l'UX Kanban et charge inutile. |
| Brancher la WS directement dans le `TaskService` | Couple la logique métier à un mécanisme de transport — exactement ce que le PO demande d'éviter. |

---

## Conséquences

**Positives :**
- Changer de techno temps réel = implémenter `Broadcaster` ailleurs et changer une ligne dans `main.go`.
- L'`EventBridge` est le seul point de traduction event domaine → message WS. Pas de logique de broadcast éparpillée.
- Le scoping par projet est enforced par le broadcaster lui-même (clés `sync.Map`), pas par chaque handler.

**Négatives / compromis :**
- Le token en query param est logué côté serveur et côté proxy — acceptable pour ce pilote, à remplacer par un cookie httpOnly ou un ticket à usage unique en production.
- `sync.Map` a ses limites à très grande échelle (pas de pub/sub distribué) — il faudra passer à Redis/NATS quand plusieurs instances de l'API coexisteront.

**Points de vigilance :**
- Vérifier que tout nouvel event métier utile au front implémente `ProjectID() string` avant d'être ajouté à la liste du bridge.
- Le `CheckOrigin: true` permet toutes les origines — restreindre en production.
