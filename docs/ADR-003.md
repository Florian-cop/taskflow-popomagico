# ADR-003 : Fiabilite de publication des domain events sans broker externe

---

## Statut

Accepté

---

## Contexte

Le fonctionnement de TaskFlow repose sur les evenements metier (`task.created`, `task.moved`, `task.assigned`, `member.added`).
Une perte d'evenement peut entrainer une incoherence fonctionnelle visible (notification absente, vue temps reel incomplete, audit incomplet).
Le systeme doit conserver un decouplage strict : un use case metier ne doit pas dependre de la reussite d'un handler specifique.
L'utilisation d'un broker externe n'est pas possible dans notre contexte actuel.
Nous ne souhaitons pas utiliser le pattern outbox.
Nous devons donc definir une strategie interne qui limite au maximum les pertes, sans ajouter de nouvelle brique d'infrastructure.

---

## Décision

Nous avons décidé d'utiliser un bus d'evenements interne (in-process) avec gestion de reprise locale des erreurs de consommation.

Concretement :
- Le use case applique d'abord la modification metier en base.
- L'evenement est ensuite publie sur le bus interne.
- Une erreur dans un handler ne fait pas echouer le use case deja valide.
- Chaque handler est isole (`try/catch`) et les echecs sont journalises dans un stockage persistant pour reprise.
- Un worker asynchrone rejoue les echecs avec backoff exponentiel.
- Chaque evenement porte un `eventId` unique ; tous les handlers sont idempotents pour tolerer les rejoues et doublons.

Le niveau de garantie retenu est un **best-effort renforce** : non-blocage metier + reprise automatique des echecs de consommation.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| RabbitMQ / Kafka / broker externe (messagerie) | Non retenu car l'usage d'un broker externe n'est pas possible dans le contexte actuel. |
| Bus interne sans retry ni journalisation | Rejete car trop fragile : une erreur de handler ferait perdre l'effet attendu sans possibilite de reprise. |
| Outbox pattern transactionnel | Non retenu par choix de conception du projet. |

---

## Conséquences

**Positives :**
- Aucune dependance a une infrastructure de messaging externe.
- Decouplage fort entre producteurs metier et consommateurs.
- Les echecs de handlers n'interrompent pas la logique metier principale.
- Les retries locaux reduisent l'impact des erreurs transitoires de consommation.

**Négatives / compromis :**
- Sans broker externe et sans outbox, une garantie mathematique de perte nulle n'est pas prouvable dans tous les cas extremes entre commit DB et publication.
- Le retry local ajoute une complexite applicative (journal d'echecs, worker, supervision).
- L'absence de stream durable natif limite les capacites de rejeu global et d'audit complet.
- Les doublons restent possibles lors des retries et imposent l'idempotence des handlers.

**Points de vigilance :**
- Imposer l'idempotence de tous les handlers (notification, websocket, audit).
- Tracer systematiquement `eventId`, `eventName`, `aggregateId`, `occurredAt` dans les logs.
- Superviser le journal des echecs et definir un seuil d'alerte clair.
- Ajouter des tests d'integration couvrant panne handler, retry et non-regression metier.
