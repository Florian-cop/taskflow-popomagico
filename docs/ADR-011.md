# ADR-011 : Versioning API par URL avec coexistence v1 / v2

---

## Statut

Accepté · remplace partiellement ADR-004

---

## Contexte

Disruption #2, message 2 : un partenaire mobile va consommer notre API et ne pourra pas suivre nos évolutions au rythme du frontend interne. Il faut donc deux versions de l'API qui **coexistent**, et une preuve que la couche de présentation n'imprègne pas le domaine.

ADR-004 avait déjà acté en théorie un versioning par URL avec coexistence `/api/v1` et `/api/v2`. Cet ADR-011 documente l'**implémentation effective** et précise la stratégie de différenciation.

Contrainte explicite du PO : « l'objectif n'est pas de dupliquer les services, mais de montrer que l'API est une couche d'entrée distincte du domaine métier ».

---

## Décision

Nous avons décidé que les deux versions partagent **strictement les mêmes services applicatifs** (`ProjectService`, `TaskService`, etc.). Seules les couches `presentation` (handlers + DTOs HTTP) sont dupliquées :

- `api/v1/handlers/` : handlers existants, format de réponse plat (héritage Phase 1).
- `api/v2/handlers/` : nouveaux handlers, format de réponse **enveloppé** :

```json
{
  "data": <payload identique à v1>,
  "meta": {
    "apiVersion": "v2",
    "generatedAt": "2026-04-15T...",
    "count": 12          // sur les listes uniquement
  }
}
```

Le câblage dans `cmd/api/main.go` :

```go
v2ProjectHandler := v2handlers.NewProjectHandler(projectService) // même service
v2TaskHandler    := v2handlers.NewTaskHandler(taskService)        // même service

r.Route("/api/v2", func(r chi.Router) {
    r.Group(func(r chi.Router) {
        r.Use(apiMiddleware.JWTAuth(tokens))
        r.Get("/projects", v2ProjectHandler.GetAll)
        // ...
    })
})
```

Les handlers v2 sont aussi minces que ceux v1 : decode → service → encode (avec wrapping). Aucun service métier n'a été modifié, copié ou refactoré.

Les endpoints v2 implémentent un **sous-ensemble représentatif** : `projects`, `tasks`, `members`. Les endpoints `notifications`, `audit` et `admin` restent v1-only pour cette livraison — les exposer en v2 est mécanique (un fichier de plus dans `api/v2/handlers/`) si un client en a besoin.

---

## Alternatives envisagées

| Alternative | Raison du rejet |
|-------------|-----------------|
| Versioning par header `Accept: application/vnd.taskflow.v2+json` | Plus REST-orthodoxe mais beaucoup moins lisible pour debug, curl, tests d'intégration. Coût pédagogique élevé pour un gain symbolique. |
| Versioning par header custom `X-API-Version: 2` | Idem : invisible dans les logs des proxys, fragile. |
| Forcer tous les clients à migrer vers v2 et déprécier v1 | Casse le contrat avec le partenaire mobile par construction. C'est exactement ce que la coexistence évite. |
| Dupliquer les services (`ProjectServiceV1`, `ProjectServiceV2`) | Anti-pattern : deux logiques métier qui divergent silencieusement. Le PO l'interdit explicitement. |
| Faire pointer v2 sur des nouveaux endpoints uniquement (`/api/v2/things`) sans recouvrement | Ne répond pas à la demande : « v2 expose les mêmes cas d'usage ». |
| Réécrire les DTOs internes pour ajouter une `enveloppe` partagée | Pollue tous les handlers v1. La duplication contrôlée des handlers v2 est plus saine. |

---

## Conséquences

**Positives :**
- Preuve architecturale : grep des imports v2 → on ne touche que `internal/{project,task}/application`. Le domaine est immobile.
- Un nouveau client peut consommer v2 sans surprise (enveloppe documentée), pendant que le frontend interne reste sur v1.
- Ajouter une v3 plus tard = créer `api/v3/handlers/`. Aucun changement de sous-jacent.
- La différenciation `meta` permet d'introduire des champs futurs (deprecation warnings, rate limit headers, pagination) sans casser v1.

**Négatives / compromis :**
- Duplication des handlers : ~80 lignes par paire (v1+v2) pour les mêmes ressources. Acceptable car les handlers sont des thin adapters. Anti-pattern si jamais un handler devient gros — signe qu'il faut extraire vers le service applicatif.
- Pour cette livraison, v2 ne reprend que `projects` + `tasks` + `members`. Les clients v2 qui voudraient consulter notifs/audit doivent retomber sur v1 — incohérent côté API design. À aligner si v2 devient la cible principale.
- Pas de gestion automatique de la dépréciation : v1 reste indéfiniment. À introduire un `Sunset` header HTTP (RFC 8594) quand la décision sera prise.

**Points de vigilance :**
- **Interdire** toute logique métier dans les handlers v1 ou v2 : si une règle métier diverge entre v1 et v2, c'est un bug. Tout doit vivre dans `internal/.../application`.
- Les tests de contrat : il faudra à terme un test par version (v1 et v2) qui vérifie le format de réponse. Pas implémenté pour cette livraison (raccourci documenté).
- L'authentification reste partagée : un même JWT est valide sur les deux versions. Cela simplifie la migration partielle d'un client mais signifie qu'on ne peut pas révoquer les tokens v1 sans toucher à v2.
- La WebSocket et les endpoints admin ne sont **pas** versionnés. Ils ne sont pas exposés au partenaire mobile, donc le besoin n'existe pas. À documenter dans la spec API publique.
